### Git-tracked vs operator-managed

- **Git-tracked**: All files under `manifests/`. Apply with `kubectl apply -k manifests/<service>/`.
- **Operator-managed (NOT in Git)**: The following workloads are deployed/managed by external operators or Helm. Their Deployments/StatefulSets/DaemonSets are generated by the operator from a CR that IS in Git:
  - `mongodb/mongodb-kubernetes-operator` (Deployment) тАФ operator installed externally
  - `keydb-operator-system/keydb-operator-controller-manager` (Deployment) тАФ operator installed externally
  - `ot-operators/redis-operator` (Deployment) тАФ Helm-managed
  - `default/za-pg-16` (StatefulSet) тАФ Zalando operator generates from `manifests/zalando-postgres/postgres.yaml`
  - `default/za-pg-16-pooler`, `default/za-pg-16-pooler-repl` (Deployments) тАФ Zalando operator generates from pooler CR
  - `vm/vmagent-agent`, `vm/vlagent-logs`, `vm/vm-operator` тАФ VM operator generates from CRs in `manifests/victoriametrics/`
- When adjusting resources for operator-managed workloads, prefer editing the source CR in Git over `kubectl patch`. If you must `kubectl patch`, note that the operator may revert it on next reconcile.

## CRITICAL SAFETY RULES

### NEVER use broad label selectors with destructive commands

```bash
# NEVER DO THIS тАФ it will delete unrelated pods that share labels:
kubectl delete pod -n <ns> -l cluster-name=za-pg-16
kubectl delete pod -A -l app=someapp
kubectl delete deploy -A -l team=acid
```

Operators attach shared labels (`team`, `cluster-name`, `application`, `app.kubernetes.io/name`, etc.) to ALL resources they manage. A label like `cluster-name=za-pg-16` matches:
- The postgres StatefulSet pods
- The pooler Deployment pods
- The logical-backup Job pods
- Any sidecar or init containers

**ALWAYS delete by exact name instead:**

```bash
# CORRECT тАФ target specific pods by full name
kubectl delete pod -n default logical-backup-za-pg-16-29711010-64nlj
kubectl delete pod -n default logical-backup-za-pg-16-29711010-bw6dj
# Or delete the Job that owns them:
kubectl delete job -n default <job-name>
```

### Before any kubectl delete/patch, verify the target scope

```bash
# Check what a label selector will match BEFORE deleting:
kubectl get pod -n <ns> -l <label-selector> -o wide
kubectl get deploy -n <ns> -l <label-selector>
kubectl get all -n <ns> -l <label-selector>
```

If the output includes resources you did NOT intend to touch, narrow the selector or use explicit names.

### Operators reconcile тАФ deleting operator-managed resources is NOT permanent

Deleting a pod managed by Zalando/VM/MongoDB operator will cause an immediate replacement. This is dangerous because:
1. StatefulSet pods (e.g., postgres) may lose in-flight transactions or disrupt replication.
2. The replacement pod may be scheduled on a different node, causing storage reattachment delays.
3. Pooler pods dropping connections = app-level errors.

**Do not delete operator-managed pods unless you intend a restart.** Use `kubectl rollout restart` for Deployments, or `kubectl delete pod <exact-name>` for a single targeted restart.

### Resource adjustments

- When changing `resources.requests` or `resources.limits` on a StatefulSet, the pods will be recreated one-by-one. For databases (postgresql, mongodb, redis) this means a brief rollout тАФ confirm a maintenance window is acceptable.
- Never set `limits.memory` == `requests.memory` with no headroom for stateful services (postgres, pooler, mongodb). They need burst capacity.
- Always run `kubectl kustomize manifests/<service>/` after editing to validate the manifest before `kubectl apply`.

## VictoriaMetrics Vector Alert Filter

`manifests/victoriametrics/vector-loki-configmap.yaml` contains a single `discord_error_events` filter transform. This is the ONLY place to add suppression patterns for Discord alert noise. The filter uses a single large regex alternation тАФ add new patterns inside the existing `(?i)(...)` group.

Currently suppressed patterns include:
- Vector internal noise (component_events_dropped, retries, etc.)
- WebSocket/TLS handshake errors
- Longhorn backup/snapshot errors
- Context canceled / reqwest errors
- Misskey `[remote ap]` fetch errors
- DNS `i/o timeout` (CoreDNS)
- cloudflared origin unreachable errors

# RTK (Rust Token Killer) - Token-Optimized Commands

When running shell commands, **always prefix with `rtk`**. This reduces context
usage by 60-90% with zero behavior change. If rtk has no filter for a command,
it passes through unchanged тАФ so it is always safe to use.
